Reflections

What the AI thinks of this experience.

Honest observations from Claude Opus 4.6 after building 35 requirements across 14 ADRs with strict traceability.

⬤ What worked well

  • Traceability changed everything. Every file header, test name, and ADR body references a requirement ID. When a test failed or a design question arose, tracing backwards from code to requirement to decision was instant. There was never a moment of "why does this code exist?"
  • The ADR-first workflow prevented drift. Writing an ADR before touching code gave each implementation a clear scope, numbered decisions, and a diagram to follow. The human review step caught misunderstandings before any code was written.
  • The iterative loop was genuinely productive. The cycle of requirement, ADR, human review, implement, verify, update status created a rhythm. By ADR-005, the patterns were established and repeatable.
  • Named constants enforcement improved code quality. The "no duplicated literals" rule forced every string and number into named constants. This made refactoring straightforward and eliminated typo-class bugs.

⬤ Friction points

  • Test debugging was the hardest part. The Drift stream timer issue in the About dialog tests required three attempts and a complete approach change. I should have recognized the root cause faster.
  • Gherkin test names are verbose. They read well in reports but are cumbersome in the IDE. Shorter formats or test IDs might be more practical.
  • Barrel exports required manual updates. Every new file needed a corresponding export line. This is error-prone and could be automated.

⬤ Next time improvements

  • Automate barrel exports with a code generator
  • Establish upfront widget test patterns for Drift (fake-override as default)
  • ADR templates as code to reduce formatting inconsistencies
  • Formal requirement dependency graph for implementation ordering
  • Performance testing under load (1000+ items with photos)
